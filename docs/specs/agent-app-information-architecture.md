# Agent App — Information Architecture (v1)

Purpose: define how the agent-facing PWA is organized — a two-tab structure
(**Today / Morning Round-Up** for planning, **Live** for real-time execution)
with a real-time interrupt layer that sits *above* the tabs. This is the
presentation/IA decision; it reads the data model already committed
(`Lead` / `CallEvent` / `MessageEvent`) and changes no schema.

Related: `speed-to-lead-architecture.md` (the interrupt layer),
`agent-input-friction-boldtrail.md` (draft-and-approve), the native agent app
(`src/app/demo/real-estate-cockpit/native/AgentApp.tsx`).

---

## 1. Two modes, two tabs — Plan vs. Act

An agent's day has two genuinely different rhythms, and the app splits along them:

| Tab | Mode | When | Feel |
|---|---|---|---|
| **Today** (Morning Round-Up) | Plan / brief | Opened with coffee; a ~10-min ritual | Proactive, batch |
| **Live** (Now) | Act / monitor | All day, reactive | Real-time, current |

Bottom-nav tabs in the PWA. This mirrors how top agents actually work: a morning
planning pass, then reactive execution the rest of the day.

## 2. Tab 1 — Today / Morning Round-Up (Plan)

**Generated fresh each morning** (a scheduled AI synthesis) and then persists as
the day's agenda — a checklist that *fills in* as work gets done, so at 3pm it's a
progress view, not a stale briefing.

Contents:
- **"Here's your day"** — today's closings, inspections, appointments, and the
  calendar-guard conflicts (from `AgentApp.tsx` calendar guard).
- **Batch follow-ups to approve-and-send** — AI-drafted nurture emails/texts
  queued for leads that need a touch. Agent skims, tweaks, one-tap sends. Each is
  a `MessageEvent` in `DRAFT` (`aiDrafted=true`, `approvedAt=null`) until approved.
- **Sphere / re-engagement nudges** — past clients newly active, saved-search matches.
- **Market note** — the day's momentum/$-per-sqft summary, client-ready.

Reads the same `Lead` rows, grouped as "today's plan."

## 3. Tab 2 — Live / Now (Act)

The real-time operational surface, always current against the CRM:
- **Response clock** — new leads as they land, colored to the 15/30-min bands.
- **Call queue** — tap-to-dial (Twilio cell-bridge), producing `CallEvent` rows.
- **Missing-doc chases**, escalations, live status as calls connect and leads move.

Reads the same `Lead` rows, surfaced as "arriving / acting now."

## 4. Above the tabs — the interrupt layer (non-negotiable)

**The speed-to-lead interrupt must NOT live behind a tab.** A hot lead cannot wait
for the agent to *open* the Live tab — that would silently break the sub-2-minute
response DoD. So the real-time layer sits above the tab structure:

- A new lead fires a **push notification + SMS** (OneSignal / Twilio) regardless
  of what's on screen, or whether the app is even open.
- Tapping the alert **deep-links straight into the Live tab, on that lead.**
- The Live tab carries a **live badge** ("1 lead cooling") visible from the Today
  tab too, so the agent is always pulled to what's urgent.

Mental model: **push is the interrupt · Live is where you land · Today is where
you plan.** The response clock starts at `Lead.receivedAt` (webhook time) and
stops at `Lead.firstTouchAt` (first attempt — a call to voicemail counts).

## 5. One data model feeds both tabs

No schema change. The same rows are read through two lenses:

| Model | Today (Plan) | Live (Act) |
|---|---|---|
| `Lead` | grouped as "today's agenda" (nurture, follow-ups) | "arriving now" + response clock |
| `MessageEvent` | batch AI drafts to approve | outbound touches as they send |
| `CallEvent` | — | the call queue + connection status |

Agent identity resolves to `BrokerageAgent` (per the schema), so both tabs and the
broker roster read consistently.

## 6. PWA specifics

- Tabs are **bottom navigation** (thumb-reachable; agents are on the move).
- Push via **OneSignal** service-worker notifications (no native app in v1).
- Deep links: a push payload carries `leadId`; tapping routes to
  `/(agent)/live?lead=<id>` and opens that lead.
- The Today briefing is (re)generated by a scheduled job (Inngest) each morning in
  the tenant's timezone.

## 7. Naming

- Tab labels: **"Today"** and **"Live"** (short; "Live" signals real-time).
- "Morning Round-Up" is the header/title *inside* the Today tab, not the tab label.

## 8. Out of scope for v1 (fast-follow)

- **Behavioral-intelligence surface** (which listings a lead is viewing) — gated on
  what the BoldTrail API exposes; a candidate third view once confirmed.
- **Seller-side** (valuations, listing pipeline) — buyer-side first.
- A dedicated **CRM/contacts browser** — v1 is action-first, not a database browser.
