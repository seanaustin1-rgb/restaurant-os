# Brokerage Data Sources — API Landscape & Ingestion Contract

**Status:** REFERENCE. **Date:** 2026-06-30. **Author:** Claude (synthesized w/ operator + Gemini cross-review).
**Purpose:** the brokerage equivalent of the MarginEdge/Toast/Plaid groundwork — what the popular
real-estate platforms' APIs actually expose and *expect*, mapped to the brokerage spine that
**already exists** in this repo (`BrokerageAgent/Deal/LeadSpend/MarketMetric` + `src/lib/brokerage/csv-import.ts`).

> **Core finding:** the brokerage financial truth (GCI, company dollar, agent splits) lives in
> **partner-gated** back-office systems. **CSV import is the MVP spine, not live APIs** — the same way
> Plaid/manual entry carries restaurants before a clean POS API. Codex already built the CSV pipeline;
> this doc says what to wire around it.

tokens are generated from the **brokerage's** admin account (delegation), not a developer signup.

### 1a. Three data layers — and the trap to avoid

The sources don't compete; they fill **three different layers.** Conflating them is the mistake:

| Layer | What it answers | Real sources | Schema target |
|---|---|---|---|
| **1. Money truth** | GCI → company dollar → splits → **caps** | **Back office (Lone Wolf gated / Loft47 CSV-only), CSV, QBO** | `BrokerageDeal.{gci,companyDollar,agentPayout}`, `BrokerageAgent.{split,cap}` |
| **2. Production & activity** | deal volume/status, side counts, pipeline velocity, **who's tapering** | Transaction-mgmt (Dotloop, SkySlope) + CRM (**MoxiWorks**, Follow Up Boss) | `BrokerageDeal.{stage,dates}`, **agent-activity surface (the gap)** |
| **3. Aura / footprint** | mindshare, sentiment, review velocity | kvCORE/BoldTrail web engagement, Google Trends (SerpApi), Brand24, Google Business Profile | reuse `aura` module |

**The trap:** Gemini's matrix maps *"Company Financial Health → Dotloop/SkySlope."* That's only half-true and strategically dangerous. Transaction-mgmt systems hold deal **value/status/escrow**, but **not the brokerage's commission economics** — company dollar, agent split, caps. Gemini concedes it: SkySlope splits *"if logged in custom fields."* That "if" is load-bearing. **Splits/company-dollar/caps are a back-office/accounting function, not transaction management.** So Layer 1 stays on **CSV + QBO + (later) back-office** — *this is exactly why CSV remains the wedge.* Dotloop/SkySlope/Moxi are Layer 2 (production + activity), not a shortcut to the money math.

**Bonus:** MoxiWorks + Follow Up Boss are the natural sources for the **"live agent-activity surface" gap** (§5) — agent logins, CMAs, pipeline velocity, tapering — which is precisely what feeds the **Agent Cockpit** and the **retention/flight-risk** premium tier.

## 2. Priority integrations — the "expectations"

### Follow Up Boss (CRM) — the easy live win
- Base `https://api.followupboss.com/v1/`; HTTPS only. Endpoints: `people`, `deals`, `notes`, `events`, `webhooks`.
- Auth: **HTTP Basic**, API key as username, blank password. Plus registered `X-System` / `X-System-Key` on every request.
- Scope follows the key's user: **broker-level key = whole account**; agent key = only that agent's people. Use a broker key.
- Rate limit: ~250 req / 10s sliding (docs also cite 1000 / 10 min); `429` on breach. Paginate + backoff.
- **Mirrors the MarginEdge `X-Api-Key` + paginate + 1 limit pattern** — cheapest brokerage API to stand up.

### Lone Wolf (back-office) — the high-value partnership
- 2025 [API Portal](https://www.lwolf.com/api-portal) across Foundation / Back Office / Transact. OAuth client creds.
- Highest-value live source (commission/financial), but BD-gated: needs a **brokerage client formally requesting access** + vendor agreement. Treat as Phase 3, unlocked with a pilot broker's leverage.
- ⚠️ **Open question to verify before relying on it:** does the API expose **commission/split line items**, or only transaction metadata? Determines whether it *replaces* CSV or only augments it.

### RESO Web API (MLS) — the market-shift layer
- REST + **OData v4** (`$filter`/`$top`/`$orderby`), **JSON**, modern auth+TLS. Data Dictionary 2.x baseline (RETS dead 2024-12-31).
- Access is **per-MLS**: the target broker signs the local MLS's Broker Data Access Agreement; you get creds from that MLS's software vendor. Standardized shape, fragmented onboarding.

### "Aura" layer — don't wait on real-estate platforms
- **Reuse the existing `aura` module** (Google/Yelp already live in this product) for the reputation value-signal.
- Optional add: social-listening (Brand24 / SerpApi) for brand-name local search velocity. Open, self-serve.

## 3. MVP build order (both AIs + operator converge here)

**Phase 1 — Core metrics (open + CSV, no partnership checks):**
- **CSV import** → `BrokerageDeal` + `BrokerageAgent` (GCI, company dollar, splits). *Already built.*
- **Follow Up Boss** → live agent roster + pipeline/lead activity.
- **QuickBooks Online** → cash-flow / OpEx ground truth to contrast vs CSV gross revenue.
- **Google Business Profile (`aura`)** → reputation signal. *Already built.*
> This covers ~90% of production + financial-health + brand-aura **without writing an enterprise check.**

**Phase 2 — Market shift (low/medium friction):** RESO Web API (broker signs MLS agreement) → `BrokerageMarketMetric`.

**Phase 3 — Automation & scale (high friction):** Lone Wolf API Portal + BoldTrail API — replace manual CSV with live sync, unlocked using a pilot brokerage's leverage.

## 5. The real gap to blend CSV + live Follow Up Boss

The schema is **not** the missing piece — these two are:

1. **Provenance + identity resolution.** `externalAgentId` is `@@unique([restaurantId, externalAgentId])`. A
   CSV export and FUB use **different ids for the same human**, so a naive blend creates **two agent rows per person.**
   Need: (a) a `sourceSystem` discriminator (`CSV | FOLLOW_UP_BOSS | LONE_WOLF | SKYSLOPE | BOLDTRAIL`), and
   (b) an identity-match step keyed on **email** (`BrokerageAgent.email` exists) to collapse one human's
   per-system ids into one canonical agent. Decide field-level authority: CSV authoritative for $ (GCI/splits),
   FUB authoritative for activity/roster.
2. **Live agent-activity surface.** FUB's value is *velocity* (active leads, pipeline movement per agent). The
   schema models closings (`BrokerageDeal`) and ad spend (`BrokerageLeadSpend`) but not a live activity rollup.
   Either reuse `BrokerageDeal` pipeline stages or add a light per-agent activity snapshot.

## 6. Product tiers & per-agent monetization ("Cockpit")

Naming locked as the two product tiers (encode the business model directly):

| Tier | Name | Who uses it | Role | Monetization |
|---|---|---|---|---|
| Leadership | **Executive Cockpit** | broker / owner / manager | the **wedge** — macro vitals, Deal-Health-vs-Ledger-Health, company-dollar retention, market + aura | flat by brokerage size band (cheap first yes) |
| Per-agent | **Agent Cockpit** | each agent (read-only personal scoreboard) | the **expansion** — own GCI, **cap progress**, pipeline, lead ROI, on-track | **per active agent / mo** (the headcount-scaling MRR engine) |
| Premium | Retention analytics | leadership-only | agent-contribution ranking, comp-plan vs company dollar, **flight-risk** | per-brokerage or per-agent uplift |

**Why per-agent works here:** brokerages are cost-sensitive on *overhead* but spend freely on **recruiting/retention** (their competitive war). The Agent Cockpit is an "every agent gets a personal cockpit" *benefit*, not an admin cost — so it sells into the broker's biggest budget, and per-agent pricing is legitimate because every seat is an active user. Cap-cliff math ("92% to cap → here's your post-cap take-home") doubles as the broker's recruiting pitch. Revenue scales with the customer's agent count, which grows as they recruit → **you grow when they grow.**

**Guardrails:** (1) Agent Cockpit ships *after* the Executive Cockpit proves value (more product: agent role + per-`agentId` scoped reads, mirrors FUB agent-key scoping). (2) Both obey the **zero-input rule** — read-only scoreboards derived from the CSV+API spine, never a place anyone types data. (3) Per-agent $ stays in leadership/agent views, **never the investor view** (agent-level profitability to outside investors = the notes-audience leak problem at person granularity). Billing meter already exists: `BrokerageAgent.status` = active count.

**Open branding question:** "Executive Cockpit" / "Agent Cockpit" are the brokerage *view/tier* names. Whether "Cockpit" also becomes the umbrella product banner (vs. staying under **OutFront Data**, live at www.outfrontdata.com) is an operator branding call — not assumed here.
